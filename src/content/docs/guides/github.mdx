## What is Git?

Git is a widely-used **distributed version control system**, created by [Linus Torvalds](https://github.com/torvalds), who is also the creator of Linux, in 2005. It's used by software developers to back up codebases, track and review changes, and work collaboratively within teams.

Codebases can be complex collections of maybe hundreds or thousands of files that reference each other in varying ways. A version control system like Git is pretty much essential when multiple people are working on the same codebase at the same time, since it records who did what, when they did it and whether there are any conflicting changes that need to be resolved.

## How Git works

Git works by storing version history in a **repository** or 'repo'. This repository can be entirely local to the developer's computer, or synced to a remote online copy using a service such as GitHub. A repository can have multiple **branches**. Each branch contains a version of the codebase that can be used for a specific purpose: for example, to create new feaures, test updates, fix bugs in isolation, or try out alternative ways of doing things. Branches can be be merged into one another to add the changes from one branch to another. The primary branch is usually called the `main` branch, but naming is flexible. In addition to branches, **tags** are often used to keep track of version numbers for different releases. More complex projects often have many branches: for example, the popular website builder, [Astro has over 400 branches and 2900 tags!](https://github.com/withastro/astro)

Saving changes to a branch, is called a **commit**. This is a bit like when you save your progress to a slot in a video game. 

Commits should at least have a summary message, but can also have a more detailed description about what has changed. There are different conventions for writing commit messages, but no universal standard. One popular way is begin with a verb in the present tense and imperative mood - e.g. `Add about.html` or `Fix styling issue with dropdown`. Another more complex specification is [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/).

The git repository data is saved to a hidden directory inside your project directory (visible if you enable e.g. 'show hidden files'). If you delete this directory, the repository will no longer exist locally. 

## Git in the terminal

You can use Git directly from the command line or terminal if you want to, providing it is insalled on your computer. Many versions of MacOS come with a version of Git pre-installed: type `git version` or `git -v` into the terminal to check if you have it!

<Steps>

1. Create a new folder called `my-git-project`.

2. Open the project folder in VS Code.

3. Open the terminal in **VS Code** (Terminal > New Terminal).

4. Use the terminal to create a new `readme.md` file:

    ```bash
    touch readme.md
    ```

5. Check git is installed on your machine:

    ```bash
    git -v
    ```

6. Initialise a new git repository:

    ```bash
    git init
    ```

7. Edit the readme file in VS Code (e.g. add `# Project Information` at the top to create a heading).

8. Stage all files for commit (there's only one file at the moment):

    ```bash
    git add .
    ```

9. Commit the changes to the readme file with a message (summary):

    ```bash
    git commit -m "Initial commit"
    ```

10. Check if there is anything further to commit (there shouldn't be):

    ```bash
    git status
    ```

</Steps>

Congratulations! You've used Git in the Terminal!

## GitHub Desktop
[GitHub Desktop](https://desktop.github.com/download/) is an optional app that allows you to manage your local and remote Git repositories from an easy-to-use GUI. While you can also manage your Git repositories directly in VS Code using the sidebar GUI (also avoiding the need to memorise terminal commands), you may still prefer the simpler and separate experience of GitHub Desktop.

- After connecting your GitHub account, you can **push** your repository changes to the online GitHub 'origin', where others can view your code.
- If your changes are on a sepearte branch, they can then be reviewed and **merged** into another branch by yourself or another collaborator. You can prompt a collaborator to do this by creating a **pull request**.
- You can also **fetch** updates others have made, and review them before **pulling** them into your branch.

### GitHub Desktop: cloning a repo:
We can clone  (create our own local copy of) a repository by choosing File > Clone Repository from the GitHub Desktop menu. If you have your own repositories on GitHub, you will be able to select them to clone here. If you want to clone someone else's repository, you can enter the URL instead.

Let's clone the following repo from GitHub: [https://github.com/ctlnorwich/portfolio-example](https://github.com/ctlnorwich/portfolio-example)

Now we've cloned our repo, we can open it in our file explorer or edit it in VS Code. Let's open it in VS Code and make a change.

- On the top toolbar, you can change your repository, change or create new branches, and fetch updates for your repo.
- In the left sidebar under the Changes tab, you can see any files that have been changed, added or deleted since the last commit.
- In the left sidebar under the History tab, you can step back and see all the file changes relating to different commits over time.
- In the main workspace, you can see the code changes for the currently selected file. This is known as the **diff**. You will notice it is colour coded to show additions and deletions.

When we're ready to submit our changes, we can commit them from the bottom-left hand corner. It's good practice to be clear and as descriptive as possible when writing commit messages.

## Forks

If we've cloned a repo we don't have permission to make changes to, we have the option to create a fork. A Fork creates a new version of the entire repository that shares code and visibility with the original. With a fork, we can use someone else's repo as a starting point for our own ideas. You have the option to preserve a connection 'upstream' back to the orignial repo, either to pull new features from the original or create pull requests from your fork for the orignal owner to review.
